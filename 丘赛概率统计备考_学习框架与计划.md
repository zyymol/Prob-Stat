# 第一部分：工作流分析与框架设计

## 1.1 mattpocock/skills 仓库关键流程节点

Matt Pocock 的 Skills 仓库（GitHub 122k⭐）围绕 AI 辅助工程实践构建了一套完整的工作流体系。其核心可拆解为以下**七个关键流程节点**：

| 编号 | 流程节点 | 核心机制 | 原始用途 |
|------|----------|----------|----------|
| N1 | **技能拆解**（Skill Decomposition） | 将复杂实践拆为单一职责小技能；区分编排层（user-invoked）与纪律层（model-invoked）；共享原语复用 | 将软件工程拆成 grill/tdd/implement/code-review 等原子技能 |
| N2 | **盘问对齐**（Grilling Session） | `/grill-me` 和 `/grill-with-docs`：AI 通过密集提问，逼迫用户澄清需求边界、暴露隐藏假设 | 消除"Agent 没做我想要的"对齐失败 |
| N3 | **共享语言**（Shared Language） | `CONTEXT.md` + `GLOSSARY.md`：建立术语表，确保每次对话使用同一套词汇体系 | 消除冗长表达，统一命名 |
| N4 | **反馈循环**（Feedback Loop） | Red-Green-Refactor：先写失败测试 → 修复通过 → 重构优化；反馈率是速度限制 | 消除"代码不工作"的盲飞 |
| N5 | **垂直切片**（Vertical Slice） | `/to-issues` 将计划分解为**端到端可交付的独立单元**，每片独立有价值 | 避免任务过大导致卡死 |
| N6 | **状态持久化**（State Persistence） | `MISSION.md` + `learning-records/` + `lessons/`：将学习状态序列化到文件系统，实现跨会话记忆 | 解决 AI"每次从零开始"的失忆问题 |
| N7 | **两轴审查**（Two-Axis Review） | `/code-review`：Standards 轴（编码标准）与 Spec 轴（忠实原始需求）**并行**审查，互不污染 | 消除架构腐化 |

此外，`/teach` 技能体现了三层学习理论：**Knowledge（事实性知识）→ Skills（动手能力）→ Wisdom（场景判断力）**，并基于维果茨基的**最近发展区（ZPD）**理论动态调整教学难度。

## 1.2 各流程节点对丘赛概率统计备考的适用性评估

### N1 技能拆解 — ✅ 高度适合

**理由**：丘赛考纲可明确拆解为「概率论模块」与「数理统计模块」两大编排层，下辖约15个子领域（收敛模式、条件期望、鞅、MLE、UMVUE、N-P引理……）。每个子领域可进一步拆为「概念理解→定理证明→真题应用」三个纪律层原子单元。这与 Skills 仓库的分层架构完全同构。

**适配改造**：将"user-invoked 编排层"映射为「周计划→日计划→复盘」的调度层；"model-invoked 纪律层"映射为「精读→推导→刷题→检验」的执行层。

### N2 盘问对齐 — ✅ 适合，但需改造

**理由**：数学备考中，"对齐失败"表现为——自以为理解了某个定理，但实际只会套公式，无法独立完成证明。盘问机制可转化为**自我诘问**：每学完一个定理，要求自己回答"这个定理的前提条件是什么？去掉哪个条件会不成立？给出反例。"

**适配改造**：将"Grilling"从"AI 追问用户"改为"用户自问自答+真题检验"。具体形式为每章学完后做一份「定理审计表」，列出：定理陈述、关键假设、去掉假设的反例、典型应用场景。

### N3 共享语言 — ✅ 高度适合

**理由**：概率统计有大量术语在不同教材中定义略有差异（如"相合性"vs"一致性"、"几乎处处收敛"vs"以概率1收敛"）。建立 `GLOSSARY.md` 可确保学习中不因术语混乱浪费时间。真题中出现的"Radon-Nikodym 导数""全变差距离""Dirichlet过程"等术语，必须提前建立精确定义。

**适配改造**：创建 `术语表.md`，按模块组织，每条包含：中文术语、英文术语、精确定义、所在教材页码、真题出现年份。

### N4 反馈循环 — ⚠️ 部分适合，需重新定义"红-绿-重构"

**理由**：Red-Green-Refactor 是软件工程的测试驱动开发循环，直接照搬到数学学习中不成立——数学没有"编译器"告诉你对错。但其核心理念"**反馈率是速度限制**"完全适用。

**适配改造**：重新定义三阶段为——
- **Red（红）**：尝试独立做一道真题/教材习题，暴露知识盲区
- **Green（绿）**：对照答案/教材，填补盲区，直到能独立写出完整证明
- **Refactor（重构）**：提炼该题的证明套路，归入「证明模式库」

### N5 垂直切片 — ✅ 高度适合

**理由**：传统学习按章节线性推进（先看完所有概率论再看统计），容易前学后忘。垂直切片要求每片"端到端可交付"——映射到备考中，意味着**每个学习单元应包含：概念→定理→习题→真题关联**，而非单纯看完一章教材。

**适配改造**：将学习计划按「知识切片」而非「教材章节」组织。每个切片以一道真题或一类真题为目标，逆向拆解所需的概念和定理。例如："2022年真题第3题（骰子模式首次出现期望）"这个切片，需要拆解出：Markov链定义、停时、首达时间方程、Wald恒等式。

### N6 状态持久化 — ✅ 高度适合，是本框架的核心

**理由**：这是 Skills 仓库中最有价值的机制。传统备考的失败模式是"学了后面忘前面，不知道自己哪里薄弱"。将学习状态序列化到文件系统，可实现跨周、跨月的持续追踪。

**适配改造**：建立完整的文档架构——
- `MISSION.md`：备考动机、目标分数、时间线
- `GLOSSARY.md`：术语表
- `learning-records/`：每日学习记录（学了什么、做了什么题、哪些不会）
- `error-bank/`：错题本（按知识点分类）
- `proof-patterns/`：证明模式库（提炼的证明套路）
- `mock-exams/`：真题检验记录

### N7 两轴审查 — ✅ 适合，改造为「双维度自检」

**理由**：数学备考也需要两个维度的审查——
- **Standards 轴（标准）**：证明是否严谨？每一步是否有充分理由？是否符合丘赛阅卷标准？
- **Spec 轴（覆盖）**：是否覆盖了考纲要求的所有知识点？真题中的高频考点是否都练过？

**适配改造**：每次阶段性检验后，做两份独立报告——「证明质量审查」和「知识点覆盖审查」。

### /teach 的三层学习理论 — ✅ 高度适合

| 层级 | 原始定义 | 丘赛映射 |
|------|----------|----------|
| Knowledge | 事实性知识（可从书上学） | 定理陈述、定义、公式（茆诗松/C&B 正文） |
| Skills | 动手能力（必须练） | 独立完成证明、推导统计量分布、构造检验 |
| Wisdom | 场景判断（需真实场景） | 判断某道真题该用哪个工具、选择最优证明路径 |

**关键洞察**：工科背景的典型瓶颈是停留在 Knowledge 层（看书看懂了），但 Skills 层严重不足（独立做不出证明）。计划必须强制分配大量时间给 Skills 层练习。

### /teach 的 ZPD 理论 — ✅ 高度适合

最近发展区理论直接指导每日学习内容的难度选择：太简单（已会的）浪费时间，太难（远超当前水平）导致放弃。计划应按渐进难度排列，并在每个切片开始前进行水平自测，动态调整。

## 1.3 专用学习迭代流程框架

基于上述批判性分析，设计如下框架——

### 框架名称：YPS-LIF（Yau Probability & Statistics Learning Iteration Framework）

### 核心理念

> **反馈率是速度限制。**（The Pragmatic Programmer）
> **每天投资于系统的设计。**（Kent Beck）
> **最好的模块是深的。**（John Ousterhout）

数学备考的等价表述：每天推进的知识切片必须有即时验证反馈；每天结束时有结构化复盘；每个知识模块应"深"——不仅会做题，还理解其在整个概率统计体系中的位置。

### 架构总览

```
YPS-LIF 架构
├── 编排层（周级调度）
│   ├── W-Plan:  周日制定本周计划（参照月度路线图）
│   ├── W-Review: 周六复盘（知识点覆盖检查 + 错题归档）
│   └── W-Plan:  周日 AI 根据复盘结果，制定下周每日 D-Study 和 D-Practice 具体内容（用户确认后执行）
│
├── 执行层（日级循环）
│   │   ┌──────────────────────────────────────────────────────────────┐
│   │   │  学习期 (Phase 0-3) 和 冲刺期 (Phase 4-5) 使用不同循环结构    │
│   │   │  各阶段时间弹性 ±0.5h，由 W-Plan 按当日难度设定               │
│   │   └──────────────────────────────────────────────────────────────┘
│   │
│   │  学习期 (Phase 0-3)：
│   ├── D-Recall   (0.5h):  AI 按间隔重复规则推送回顾内容（昨日/3天前/7天前/模考失分）
│   ├── D-Study    (1-1.5h): AI 讲解→用户确认"懂了"→AI 出题；未懂则不出新题
│   ├── D-Practice (1.5-2.5h): AI 沉默；用户独立做题
│   ├── D-Verify   (0.5h): AI 批改→用户消化5-10min→一句话总结教训
│   └── D-Refactor (0.5h): 用户写笔记→AI ingest 更新 wiki
│   │
│   │  冲刺期 (Phase 4)：做题4h→批改0.5h→消化0.5h→Ingest 0.5h
│   │  查漏期 (Phase 5)：D-Recall 0.5h→D-Practice 2.5h→D-Verify 0.5h→D-Refactor 0.5h
│
├── 检验层（阶段级，每 3-4 周一次）
│   ├── S-Mock:   掐表做真题套卷（4小时，模拟竞赛）
│   ├── S-Audit:  双维度自检（证明质量 + 知识覆盖）
│   └── S-Pivot:  计划调整（增减时间分配、调整难度）
│
└── 持久化层（kb-wiki 文件系统）
    ├── AGENTS.md          ← Schema（生产规范）
    ├── wiki/index.md      ← 16集群概念页主目录
    ├── wiki/hot.md        ← 会话热缓存（ZPD上下文）
    ├── wiki/concepts/     ← 概念页（存用户理解状态，非教材原文）
    ├── wiki/syntheses/    ← 证明模式库
    ├── wiki/log.md        ← 活动日志
    └── raw/notes/         ← 用户每日学习笔记（ingest 输入）
```

### 衔接机制

| 衔接点 | 机制 | 输入 → 输出 |
|--------|------|-------------|
| 日→日 | D-Refactor 的 ingest 更新 wiki/hot.md → 次日 D-Recall 读取 | 昨日错题+理解状态 → 今日回顾清单 |
| 日→周 | 每日 ingest 的 wiki/log.md 条目汇入 W-Review 的 lint | 7天概念页 confidence 变化 → 周度复盘报告 |
| 周→周 | W-Plan：AI 根据本周 lint 报告制定下周每日内容 | confidence 趋势 → 下周每日 D-Study/D-Practice 具体内容 |
| 周→阶段 | 每 3-4 周触发 S-Mock | 累积学习状态 → 真题模拟 |
| 阶段→阶段 | S-Pivot 重校准路线图 | 模考反馈 → 后续计划修订 |
| 阶段→总目标 | 最终 S-Mock 系列（冲刺期） | 全部真题 → 竞赛状态评估 |

### 周期设计

| 阶段 | 周数 | 核心任务 | 检验节点 |
|------|------|----------|----------|
| **Phase 0: 地基** | 第1-5周 | 工科概统复习(1周) + 测度论补课(3周) + 综合(1周) | S1: 第5周末，测度论基本概念测试 |

# 第二部分：详细学习计划

## 2.1 宏观路线图（5+6+6+6+6+4 = 33周）

### Phase 0: 地基补齐（第1-5周，125小时）

> **目标**：先复习工科概统打基础，再补齐测度论天堑，最后整合两者。
> **核心教材**：茆诗松第1-3章（复习） + 程士宏《测度论与概率论基础》前半部分
> **ZPD 判断**：第1周为复习，难度适中；第2-4周测度论概念抽象，每天控制在4小时。

### Phase 1: 概率论深度攻坚（第6-11周，150小时）

> **目标**：掌握 Durrett 核心内容——收敛模式、条件期望、鞅、随机过程。
> **核心教材**：Durrett 第1-6章 + 茆诗松第4-5章
> **检验**：第11周末做2020年真题

### Phase 2: 统计推断理论（第12-17周，150小时）

> **目标**：掌握 C&B 核心内容——充分性、估计理论、假设检验、贝叶斯、大样本。
> **核心教材**：C&B 第6-10章 + 茆诗松第6-8章
> **检验**：第17周末做2018-2019年真题

### Phase 3: 拓展攻坚（第18-23周，150小时）

> **目标**：补齐真题超纲点——决策论/Minimax、渐近统计/M-估计、耦合/TV距离、非参数Bayes/Dirichlet过程、随机图/随机矩阵。
> **核心教材**：van der Vaart《Asymptotic Statistics》前5章 + Keener第6-7章 + Ferguson(1973)论文
> **检验**：第23周末做2020-2021年真题

### Phase 4: 真题冲刺（第24-29周，150小时）

> **目标**：系统刷完所有年份真题，建立证明模式库，训练竞赛节奏。
> **核心素材**：全部真题（2012-2023），按年份和专题两遍交叉刷
> **检验**：第29周末做2022-2023年真题（模拟实战）

### Phase 5: 查漏补缺（第30-33周，100小时）

> **目标**：回炉薄弱点，全真模拟，调整心态。
> **检验**：第33周末全真模拟一套（6小时掐表）

**总学时**：约825小时（33周 × 平均25小时/周）

## 2.2 逐周逐日计划

> **每日时间模板（学习期 Phase 0-3，约5h，各阶段弹性±0.5h）**：
> - 0.5h D-Recall：AI 读 hot.md，按间隔重复规则推送回顾内容
> - 1-1.5h D-Study：AI 讲解知识点 → 用户确认"懂了" → AI 出 2-3 道分层题
> - 1.5-2.5h D-Practice：**AI 沉默**，用户独立做题（时长由 W-Plan 按题目难度设定）
> - 0.5h D-Verify：AI 批改 → 留 5-10min 用户消化 → 用户一句话总结教训
> - 0.5h D-Refactor：用户写笔记 → **AI ingest** 更新 wiki
>
> **冲刺期模板（Phase 4 真题冲刺，约5.5h）**：
> - 4h 做题：掐表做真题套卷，AI 沉默
> - 0.5h 批改：AI 逐题批改 + 标准答案
> - 0.5h 消化：用户对照理解错误
> - 0.5h Ingest：用户写错题反思 → AI ingest
>
> **查漏期模板（Phase 5，约4h）**：
> - 0.5h D-Recall：AI 推送 lint 报告的薄弱概念
> - 2.5h D-Practice：重做错题 + 专项练习
> - 0.5h D-Verify：AI 批改 + 确认掌握度
> - 0.5h D-Refactor：ingest 更新 confidence

### Phase 0: 地基补齐（第1-5周）
